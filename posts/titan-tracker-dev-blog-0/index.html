Titan Dev Diary Entry #1

21 Feb 2022

[Project Repo|https://github.com/ianbruyere/titan-army-tracker]

Background

Titan(1980) is a fun board game, but has significant flaws.
Some flaws are part of the charm, however some lead to unwieldy games.
I want to reduce some of the unwieldliness with digital aid.
Mainly to make moving armies easier, and distribute public army information without having to ask an opponent constantly.
Some fringe benefits would make saving game state easier since Titan can last upwards of 4 hours,
takes up significant table real estate, and moving it while preserving game state is an arduous task.

Day -1

I some false starts where I wanted to make a phone app for it.
This has some obvious benefits, since phones take up less space than laptops since the board is still in play, this matters quite a bit . However, I’ve never made a phone app, and the ambition of this project opened it’s maw and swallowed me whole on more than one occasion.
I finally decided to simply make it in Python, since I could figure out logic, layouts, and how it would be organized without having to navigate
unfamiliar xml layouts and the android API. With that settled I had rekindled motivation(and confidence) .

Day 0

I started with some online resources to get my head in the right place. Resources linked below.
My Google query was something along the lines of “Game Session with Python”.
I’d never done GUI or network programming with Python; I was on familiar ground syntax-wise, but these were libraries I’d never touched.
I didn’t even know Python had an in-built GUI library in the form of tkinter.
Using the blog post linked below as a guide, I did my standard ad hoc remix session:

  • skim the post, mainly looking at code
  • start writing out the code, changing names where I see fit, noting where something doesn’t apply
  • a lot of start-stop-read-repeat going on.
  • stubbing out other things I think I’ll need, even if it’s just an empty file, with a name I think fits

End result: I had the Client/Server code in the Blog post written out, and a skeleton of directories and functions that would guide me on the morrow.
Nothing worth showing off on git-hub, but at least a start. This was maybe about as far as I had gotten on the phone app version, but in a couple hours
as opposed to multiple sessions over multiple days. Plus I managed to do it off of one post, since my abilities to evaluate Python resources is much higher
then for Android Studio. [Blog Post|https://levelup.gitconnected.com/program-a-networked-tic-tac-toe-game-in-python-30f8826e591d]

Day 1

Commit: 18a0958c306d60c9e993f02013fcda2be542b9e2
Now I’m on my own.
The blog post, while useful as an occasional reference, I was far beyond it’s scope.
I floundered for a bit on where to start. Mainly stubbed out some code for future reference. I decided figuring out the client layout and how tkinter passed things around was important so I worked on the Setup Screen. The setup screen is for a player to input starting info name, team color, starting army icons etc.
What a session this turned out to be.
I made headway quite quickly, getting the dropdowns where I wanted, and loading the army icon names into them quite quickly.
Getting the image, or rather an image, to load in didn’t take too long either.
With the basic elements in place, the stage was set for trouble to manifest.
I cut the heads of each problem, but failed to cauterize, as each resolution would cause a new head to spring up:

  • separate drop downs when one changed the other would change to match it
  • image wouldn’t change in response to dropdown changing
    • during callback event, label information wasn’t present –> major sticking point
  • general UI plague

It was about a 6 hour effort this time. I wasted time by failing to note that SO answers were 8+ years old referring to Python 2 tkinter implementation.
This was hamstringing me in one specfic way, and compounded with my general unfamiliarity with tkinter caused me quite a few headaches.
Refactoring my code helped greatly though, reducing the surface area of where errors could be coming from and allowing me to find problem spots much easier.
Though intially I was post-poning it for after I’d resolved the issues.
I was able to have a complete Setup Screen to show off.

Day 2

Commit: ad3e49bb532ebca99a332defa2a84e9b230f9e00
This was the day where I bit off more than I could chew. Or at least I felt that way. When I don’t fully understand something, it can feel daunting.
I wasn’t sure how tkinter was behaving or would behave between multiple window states.

  1. How would the information flow between them?
  2. How would I maintain state of the other players?
  3. How would each player get their own GUI element in the same app but be able to update discretely?

I’m good at coming up with the questions I need answered, but finding adequate answers is troublesome.
The blog circuit has lost it’s utility at this stage: it too heavily focuses on toy examples, and also more on how than why or whats going on.
Typically at this point I RTFM but it’s more of a problem of what I was using tkinter for, rather then with tkinter specifically.
Most of my time today I spent thinking about the problem as I took care of other concerns: chores, working out, being a bum(it was a day off after all).
I’d sit down to code intermittenly as thoughts crystallized, but paralysis would soon strike.
My attempt to gain traction was wrongheaded or the hugeness of what I had to do/figure out would loom large.
What eventually helped was my knowledge of web programming. I’ve done frontend stuff before; I was only unfamiliar with tkinter, not the concepts it was built off.
With that mental model, I was able to hack out something that I think will be the basis of how the whole project will work.
Also worth mentioning is one of stumbling steps forward throughout the day had me simply make another tk app class.
A stopgap by skirting Question 3, wrestling the problem into something my brain can handle' despite the avoidance, I now have an idea of how I';; manage it..
Refactoring is in my future.
An anticipatory moment for me: you get feature working, spend a couple cathartic sessions reorganizing, and refactoring.
Then seeing all those changes in git…chef’s kiss. It’s almost like I know how to code.
Enjoy the spaghetti while it lasts though.
